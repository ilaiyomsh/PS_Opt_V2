1. מהו Gaussian Process?

Gaussian Process (GP) הוא מודל הסתברותי לא-פרמטרי שמגדיר התפלגות על פונקציות. במקום ללמוד פרמטרים קבועים (כמו ברגרסיה לינארית), ה-GP לומד את הקשר בין נקודות באמצעות פונקציית קרנל (Kernel).

הרעיון המרכזי: בהינתן נקודות שכבר נמדדו, ה-GP מספק שני דברים לכל נקודה חדשה:

μ(x) — ממוצע (Mean)

הניחוש הטוב ביותר של הערך בנקודה x. מבוסס על הערכים הידועים ועל המרחק מהם.

אינטואיציה: "מה לדעתי הערך פה?"

σ(x) — סטיית תקן (Std)

רמת אי-הוודאות בניחוש. קרוב לנקודות ידועות → σ נמוך. רחוק → σ גבוה.

אינטואיציה: "כמה אני בטוח?"

ויזואליזציה: Gaussian Process - ממוצע ואי-וודאות
μ(x) — ממוצע
±2σ — תחום אי-וודאות
נקודות נצפות

פונקציית הקרנל — Matérn (ν=2.5)

הספרייה משתמשת בקרנל Matérn עם ν=2.5, שמגדיר את ה"דמיון" בין שתי נקודות. ככל שנקודות קרובות יותר, הקרנל גבוה יותר, והמודל מניח שהערכים שלהן דומים.

K(x, x') = σ² · (1 + √5·r + 5r²/3) · exp(-√5·r)
כאשר r = |x - x'| / l, ו-l הוא פרמטר ה-length scale

Reference: bayesian_optimization.py:132-138 — GaussianProcessRegressor(kernel=Matern(nu=2.5))

תכונות מפתח של GP

  • בנקודות נצפות: σ(x) → 0 (אי-הוודאות כמעט אפסית)
  • רחוק מנקודות נצפות: σ(x) גבוה (אנחנו לא יודעים מה קורה שם)
  • אחרי הוספת נקודה: ה-GP מתעדכן — μ ו-σ משתנים בכל המרחב
  • normalize_y=True — המודל מנרמל את ערכי ה-target לפני הלמידה

2. פונקציית הרכישה — Upper Confidence Bound

פונקציית הרכישה (Acquisition Function) היא הכלי שמחליט איפה לדגום הבא. היא מאזנת בין שני שיקולים מתחרים:

Exploitation — ניצול

"דגום באזור שנראה טוב (μ גבוה)"

יתרון: שיפור מהיר. חיסרון: עלול לפספס אזורים טובים יותר.

Exploration — חקירה

"דגום באזור שלא נבדק (σ גבוה)"

יתרון: מוצא אזורים חדשים. חיסרון: אבוד זמן על אזורים גרועים.

ה-UCB פותר את הדילמה הזו על ידי שילוב של שניהם בנוסחה אחת:

UCB(x) = μ(x) + κ · σ(x)
הנקודה הבאה לדגימה = argmax UCB(x)
κ (kappa) שולט באיזון: κ גבוה → יותר חקירה, κ נמוך → יותר ניצול

Reference: acquisition.py:357-373 — base_acq: return mean + self.kappa * std

ויזואליזציה: UCB — איזון בין חקירה לניצול
μ(x) — ממוצע
σ(x) — אי-וודאות
UCB(x) = μ + κσ
argmax UCB — הנקודה הבאה
נקודות נצפות

ערכי κ שונים

κ התנהגות מתאים ל...
0 ניצול מלא — בוחר רק לפי μ שלב אופטימיזציה מתקדם
2.576 ברירת מחדל — איזון סביר (99% confidence interval) שימוש כללי
10+ חקירה אגרסיבית — בוחר אזורים לא ידועים שלב ראשוני, מרחב גדול

5. ה-Landscape — למה ה-GP "מתבלבל"

מהו Landscape?

בהקשר של אופטימיזציה, Landscape (נוף / משטח) הוא ה"שטח" שפונקציית ה-Cost יוצרת מעל מרחב הפרמטרים. דמיינו מפת טופוגרפיה:

מרחב הפרמטרים = "הקרקע"

כל נקודה על המפה מייצגת שילוב של פרמטרים:

(w_r, h_si, S, doping, L, λ)

במקרה שלנו, מרחב בעל 6 ממדים.

ערך ה-Cost = "הגובה"

הגובה בכל נקודה הוא ערך ה-Cost באותם פרמטרים:

  • עמקים = פרמטרים טובים (cost נמוך)
  • פסגות = פרמטרים גרועים (cost גבוה)

תפקיד ה-GP הוא ללמוד את צורת ה-landscape מתוך הנקודות שנדגמו, ותפקיד ה-UCB הוא להחליט איפה לדגום הבא על סמך ההבנה הזו.

הבעיה: טווח ערכים קיצוני

ה-Landscape "האמיתי" של הבעיה שלנו מכיל פרטים עדינים — הבדלים בין cost של 8 ל-50 שמייצגים שילובי פרמטרים שונים באיכותם. אבל ה-GP לא רואה את הפרטים האלה, כי:

אנלוגיה: הר אוורסט מול גבעות

דמיינו שאתם מסתכלים על נוף מלמעלה. יש גבעות קטנות (גובה 8-50 מטר) ויש הר ענק (גובה 1,000,000,000 מטר — מיליארד מטר!). כשאתם מסתכלים על מפת גובה שכוללת את ההר הענק, כל הגבעות נראות שטוחות לגמרי — אי אפשר להבחין בין גבעה של 8 מטר לגבעה של 50 מטר כשהסקאלה מגיעה למיליארד.

זה בדיוק מה שקורה ל-GP: ההבדל בין -8 ל--50 הוא 0.0000004% מהטווח הכולל (-8 עד -1e9). המודל פשוט לא "רואה" את זה.

מה זה עושה ל-UCB?

כש-landscape שטוח מנקודת המבט של ה-GP:

  • μ(x) כמעט קבוע — ה-GP חוזה ערך דומה (ממוצע) כמעט בכל מקום
  • σ(x) שולט לבד — אם μ שטוח, אז UCB = μ + κσ ≈ const + κσ
  • ה-argmax של UCB נקבע רק לפי σ — האופטימייזר הולך לאזור עם הכי הרבה אי-וודאות, לא לאזור עם הכי הרבה פוטנציאל
  • אם אותו אזור שומר על σ גבוה (למשל בגלל מבנה מרחבי מסוים), ה-UCB יצביע עליו שוב ושוב
ויזואליזציה: UCB על Landscape שטוח — σ שולט
μ(x) — כמעט שטוח
κ·σ(x) — שולט ב-UCB
UCB = μ + κσ
argmax UCB — תמיד אותו מקום

הלולאה האינסופית

  1. ה-GP מאומן על כל הנתונים (כולל ערכי -1e9) → landscape שטוח
  2. UCB ממקסם בנקודה X (שם κ·σ גבוה ביותר, כי μ שטוח)
  3. הסימולציה רצה → נכשלת → target = -1e9
  4. הנקודה נרשמת, ה-GP מאומן מחדש
  5. ה-landscape עדיין שטוח → UCB שוב ממקסם בנקודה X
  6. חזרה לשלב 3 — לנצח

6. זרימת חישוב ה-Cost — שני חישובים לא תואמים

במערכת קיימים שני חישובי cost נפרדים שפועלים באופן שונה. הבנת ההבדל ביניהם חיונית להבנת הבעיה.

חישוב #1: בתוך run_row() — לשמירה ב-CSV

מיד אחרי כל סימולציה, run_row() מחשב cost ושומר אותו בעמודת cost ב-CSV. 

# run_simulation.py:575-588
is_valid = not np.isnan(v_pi_l)

if is_valid:
    # Success: Eq. 27 top case
    norm_loss = (loss_at_v_pi / config.TARGETS['loss']) ** 2
    norm_vpil = (v_pi_l / config.TARGETS['vpil']) ** 2
    cost = config.FOM_WEIGHTS['loss'] * norm_loss + config.FOM_WEIGHTS['vpil'] * norm_vpil
else:
    # Failure: HARDCODED placeholder!
    norm_loss = np.nan
    norm_vpil = np.nan
    cost = 1e9  # ← Not using C_BASE at all!

הבעיה

במקרה כישלון, run_row() שומר cost = 1e9 — ערך placeholder קבוע. הוא לא קורא ל-calculate_loss_function() ולא משתמש ב-_C_BASE הדינמי.

חישוב #2: בתוך train_optimizer() — לרישום ב-GP

כשה-optimizer מאומן, הוא קורא שוב את ה-CSV, מתעלם מעמודת cost, ומחשב מחדש את ה-cost לכל שורה באמצעות calculate_loss_function(): 

# BO.py:218 — inside train_optimizer() loop
cost = calculate_loss_function(alpha, v_pi_l, max_dphi)
optimizer.register(params=params, target=cost)

הפונקציה calculate_loss_function() משתמשת ב-_C_BASE הדינמי (שחושב מ-max cost של סימולציות תקינות): 

# BO.py:364-415 — calculate_loss_function()
if is_success:
    # Same formula as run_row
    cost = w_loss * (alpha/T_loss)² + w_vpil * (vpil/T_vpil)²
else:
    # Dynamic penalty using _C_BASE!
    cost = _C_BASE + _PENALTY_BETA * (π - max_dphi)²
    # Where _C_BASE = max(valid costs), _PENALTY_BETA = 9*C_BASE/π²

return -cost  # Negative: BO maximizes, we minimize

דיאגרמת זרימה: שני חישובי ה-Cost

בעיית ה-_C_BASE המיושן

_C_BASE מחושב רק פעם אחת — כשהאופטימייזר מאומן בתחילת כל איטרציה:

מתי _C_BASE מתעדכן?

מיקום פונקציה מתי?
BO.py:140 train_optimizer() תחילת כל איטרציית BO — פעם אחת לסימולציה
BO.py:441 get_best_result() רק בדיווח — לא משפיע על ה-GP

בפועל, מכיוון שכל איטרציה של ה-BO loop קוראת ל-train_optimizer() מחדש (שורה 273 ב-main.py), ה-_C_BASE מתעדכן בתחילת כל איטרציה על בסיס כל התוצאות שנצברו עד כה. עם זאת, ה-cost שנשמר ב-CSV (1e9) לא תואם את ה-cost שה-GP מחשב בפועל (דינמי), מה שיוצר חוסר עקביות בנתונים.

7. בעיית פונקציית ה-Cost — שורש הבעיה האמיתי

גם אם נפתור את בעיית הכפילויות עם ConstantLiar, ה-GP עדיין לא ילמד כלום. פונקציית ה-Cost עצמה היא הבעיה המרכזית — היא יוצרת טווח ערכים כל כך קיצוני שה-GP לא מסוגל לבנות מודל שימושי.

הנתונים מהריצה — ערכי Cost של סימולציות "תקינות"

כל הסימולציות הבאות הגיעו ל-π phase shift — כלומר, כולן "תקינות" מבחינת פונקציית ה-Cost. אבל הערכים שלהן שונים בסדרי גודל:

טווח ערכים של פקטור 70,000,000

הסימולציות ה"תקינות" מייצרות ערכי cost בין 8.7 ל-5.6e8 — הבדל של ×70,000,000. אחרי שה-GP מנרמל את הערכים, ההבדל בין 8.7 (מצוין) ל-42 (סביר) הוא 0.000006% מהטווח הכולל. ה-GP פשוט לא רואה את זה.

למה זה קורה? — הריבוע ההרסני

cost = 0.3 × (α / 2.0)² + 0.7 × (VπL / 1.0)²
כשה-Loss הוא 68,000 dB/cm:   (68,000 / 2.0)² = 1.16 × 10⁹

הבעיה היא הריבוע ללא הגבלה. כשערכי ה-loss הם אלפי dB/cm (שקורה כש-S קטן מאוד → ספיגה מסיבית), הריבוע מפוצץ את ה-cost לערכים אסטרונומיים. סימולציות אלה "תקינות" טכנית (הגיעו ל-π) אבל חסרות משמעות פיזיקלית.

שלוש בעיות מבניות

בעיה 1: אין הגבלה עליונה (No Clipping)

ערכי cost תקינים יכולים להגיע למיליארדים. אין שום מנגנון שמגביל את הערך המקסימלי. סימולציה עם loss של 86,000 dB/cm מקבלת cost של 5.6e8 — שזה חסר משמעות מבחינת האופטימיזציה. כל מה שה-GP צריך לדעת הוא "זה גרוע", לא "כמה גרוע בדיוק".

בעיה 2: הריבוע מעוות את הסקאלה

הנוסחה (α/T)² יוצרת צמיחה ריבועית. כש-α עולה מ-10 ל-68,000 (פקטור 6,800), ה-cost עולה מ-7.5 ל-3.5e8 (פקטור 46,000,000). הריבוע הופך הבדלים ליניאריים להבדלים אסטרונומיים.

בעיה 3: "תקין גרוע" גרוע מ"כישלון"

C_BASE = max(valid costs) = 5.6e8. עכשיו שימו לב: כישלון עם Δφ קרוב ל-π מקבל penalty של 5.6e8 + β·(π-Δφ)² ≈ 5.6e8 (כי β·ε² קטן). אבל סימולציה "תקינה" עם loss גבוה כבר מקבלת 5.6e8.

כלומר: הגבול בין "תקין" ל"כישלון" מטושטש. ה-GP לא יכול ללמוד שכישלון הוא תמיד גרוע יותר מהצלחה.

פתרון C: שילוב — Log + Clipping + ConstantLiar

הפתרון המקיף ביותר משלב את כל הגישות:

  1. Clip — הגבלת cost תקין למקסימום סביר (למשל 1000)
  2. Log — דחיסה לוגריתמית של הטווח
  3. Penalty ברור — כישלון תמיד מקבל ערך גרוע יותר מכל הצלחה

def calculate_loss_function(alpha, v_pi_l, max_dphi):
    MAX_COST = 1000

    if is_success:
        raw_cost = w_loss * (alpha/T_loss)**2 + w_vpil * (vpil/T_vpil)**2
        cost = np.log1p(min(raw_cost, MAX_COST))
    else:
        # Failure: always worse than log1p(MAX_COST) = 6.91
        base = np.log1p(MAX_COST)  # ≈ 6.91

        cost = base + 3.0 * phi_gap  # Range: [6.91, 9.91]

    return -cost

# acquisition.py:816-897 — ConstantLiar.suggest()
def suggest(self, gp, target_space, ...):

    # 1. Remove dummies that have real evaluations now
    self._remove_expired_dummies(target_space)

    # 2. Create a COPY of the target space
    dummy_target_space = self._copy_target_space(target_space)

    # 3. Choose the "lie" value based on strategy
    if self.strategy == "max":
        dummy_target = target_space.target.max()

    # 4. Register all previous dummies into the copy
    for dummy in self.dummies:
        dummy_target_space.register(dummy, dummy_target)

    # 5. Fit GP on the augmented copy
    self._fit_gp(gp=gp, target_space=dummy_target_space)

    # 6. Suggest using the base acquisition (UCB)
    x_max = self.base_acquisition.suggest(gp, dummy_target_space, fit_gp=False)

    # 7. Remember this point as a new dummy
    self.dummies.append(x_max)

    return x_max

אסטרטגיות ה"שקר"

Strategy ערך ה-dummy אפקט
"max" (ברירת מחדל) הערך הגבוה ביותר שנצפה מעודד חקירה. ה-GP חושב שהנקודה כבר נבדקה ונתנה תוצאה טובה (max). σ יורד → UCB יורד → האופטימייזר מחפש במקום אחר.
"min" הערך הנמוך ביותר שנצפה מעודד ניצול. ה-GP חושב שהנקודה נתנה תוצאה גרועה, ומחפש באזורים עם μ גבוה יותר.
"mean" הממוצע מאוזן.
float ערך מותאם אישית שליטה מלאה.

למה ConstantLiar עובד ו-register(-1e10) לא?

הגישה הנוכחית (שבורה)

  • רושם ישירות ל-target space האמיתי
  • allow_duplicate_points=False → NotUniqueError
  • ערך -1e10 מעוות את המודל
  • רק 5 ניסיונות חוזרים
  • ה-GP לא מאומן מחדש בין ניסיונות

ConstantLiar

  • עובד על עותק של ה-target space
  • לא מושפע מ-allow_duplicate_points
  • ערך dummy סביר (max/min/mean)
  • כל suggest() מוסיף dummy חדש → אין כפילויות
  • GP מאומן מחדש עם ה-dummies על העותק
